# 当前后端开发计划审阅版

本计划只约束当前后端开发顺序和验收标准，不涉及前端页面实现。数据库结构以 `docs/数据库计划.md` 和 `sql/02_create_tables.sql` 为准；如果本文与数据库设计存在出入，以数据库设计为准。

## 一、当前状态判断

当前项目已经具备继续后端开发的条件：

```text
1. MySQL 数据库和 12 张表已经创建
2. Spring Boot 已能通过 JdbcTemplate 连接 MySQL
3. college 的 Entity、Dao、Service、测试已经建立
4. Maven 测试可在注入 DB_PASSWORD 后通过
```

原计划总体方向合理：先打通一个基础模块，再横向复制到其他基础表，最后做人员、登录、任课、选课、成绩和报表。这种路线比一次性写完所有 CRUD 更稳。

需要调整的地方：

```text
1. 不建议先单独花一整阶段写“通用基础设施”，应和 CollegeController 一起落地。
2. 登录权限不能放得太晚，否则后面教师、学生、选课、成绩接口会缺少身份上下文。
3. 基础表 CRUD 要有统一模板，但不要机械复制，必须保留外键存在性校验和删除前关联校验。
4. 报表接口必须进入 P0 或 P1 交界位置，因为任务书明确要求统计报表和图形分析报表。
```

## 二、后端总体架构

后端调用链保持：

```text
Controller 接收请求
    ↓
Service 处理业务规则
    ↓
DAO 执行 SQL
    ↓
JdbcTemplate 操作 MySQL
    ↓
Result 统一返回
```

包结构保持在：

```text
src/main/java/com/tzufucius/edu/edumanagementsystem
├─ common        通用返回结果、分页对象
├─ entity        单表实体
├─ dto           多表联查展示对象
├─ dao           数据库访问层
├─ service       业务逻辑层
├─ controller    接口控制层
├─ config        Web 配置
├─ interceptor   登录和权限拦截器
└─ exception     全局异常处理
```

开发约束：

```text
1. 继续使用 JdbcTemplate，不改 MyBatis、MyBatis-Plus 或 JPA。
2. 不直接硬删除业务数据，统一通过 status 做逻辑删除。
3. Service 层必须写业务校验，Controller 不直接拼 SQL。
4. 多表展示用 DTO，不把联查字段塞进单表 Entity。
5. 涉及两张以上表同时写入时，Service 方法必须使用 @Transactional。
```

## 三、修正后的阶段计划

### 第 1 阶段：打通 College 接口闭环

当前不要先复制其他表，先把 `college` 从 DAO/Service 推进到 Controller 层，形成可通过 HTTP 调用的完整模板。

本阶段实现：

```text
common/Result.java
exception/GlobalExceptionHandler.java
controller/CollegeController.java
```

College 接口：

```text
GET    /api/colleges
GET    /api/colleges/{id}
POST   /api/colleges
PUT    /api/colleges/{id}
DELETE /api/colleges/{id}
```

验收标准：

```text
1. CollegeDaoTest 通过
2. CollegeServiceTest 通过
3. Maven test 通过
4. Apifox、浏览器或 curl 可以访问 /api/colleges
5. 接口错误返回统一 Result，而不是异常堆栈
```

说明：`PageResult` 暂时不强制实现。列表数量不大时先返回 `List`，分页和模糊搜索放到后续统一增强。

### 第 2 阶段：复制基础组织表

完成最稳定、依赖清晰的基础表：

```text
major         专业
department    教研室
class_info    班级
course        课程
```

开发顺序：

| 顺序 | 模块 | 依赖 | 删除前校验 |
| --- | --- | --- | --- |
| 1 | major | college | 专业下是否还有班级 |
| 2 | department | college | 教研室下是否还有教师 |
| 3 | class_info | major | 班级下是否还有学生 |
| 4 | course | 无直接父表 | 课程是否已有任课安排 |

每个模块至少包含：

```text
Entity
Dao
Service
Controller
DaoTest
ServiceTest
```

基础接口统一为：

```text
GET    /api/{resources}
GET    /api/{resources}/{id}
POST   /api/{resources}
PUT    /api/{resources}/{id}
DELETE /api/{resources}/{id}
```

Service 必须校验：

```text
1. 必填字段不能为空
2. 编号字段不能重复
3. 外键引用对象必须存在且 status 正常
4. 删除前检查下级或业务关联
```

验收标准：

```text
1. 四个模块均可通过接口完成查、增、改、逻辑删
2. 外键不存在时返回明确业务错误
3. 有下级数据时不允许逻辑删除父级数据
4. Maven test 通过
```

### 第 3 阶段：提前完成用户与登录权限骨架

原计划把登录权限放在学生、教师之后，风险偏高。建议在基础组织表完成后，先实现 `sys_user` 和登录骨架，再开发学生、教师模块。

本阶段实现：

```text
SysUser Entity / Dao / Service
AuthController
LoginInterceptor
WebConfig
```

接口：

```text
POST /api/auth/login
POST /api/auth/logout
GET  /api/auth/current
```

登录成功后，Session 中至少保存：

```text
userId
username
role
```

拦截规则：

```text
拦截：/api/**
放行：/api/auth/login
```

角色先采用简单枚举：

```text
ADMIN
TEACHER
STUDENT
```

验收标准：

```text
1. admin / teacher / student 三类账号均可登录
2. 未登录访问业务接口会被拦截
3. 登录后可以通过 /api/auth/current 获取当前用户
4. 暂不引入 Spring Security
```

### 第 5 阶段：任课安排 `teaching_task`

任课安排是后续选课和成绩的前置模块。

依赖：


```text

DELETE /api/teaching-tasks/{id}
GET    /api/teachers/{teacherId}/teaching-tasks
```

建议 DTO：


```text

4. startSection 不能大于 endSection
5. 删除任课安排前检查是否已有学生选课
```

验收标准：

```text
1. 管理员可以创建和维护任课安排
2. 教师可以查询自己的任课安排
3. 任课安排被学生选择后不能直接删除
```

复杂排课冲突检测先放 P2，不阻塞主流程。

### 第 6 阶段：学生选课与成绩

这是项目核心业务流程，需要重点保证事务一致性。

学生选课接口：

```text
POST /api/student-courses/select
```

学生退课接口：

```text
POST /api/student-courses/drop
```

教师录入成绩接口：

```text
PUT /api/student-courses/{id}/score
```

学生查成绩接口：

```text
GET /api/students/{studentId}/grades
```

选课事务：

```text
插入 student_course
    ↓
teaching_task.selected_count + 1
```

退课事务：

```text
student_course.status = 0
    ↓
teaching_task.selected_count - 1
```

Service 校验：

```text
1. 学生存在且状态正常
2. 任课记录存在且可选
3. 容量未满
4. 不能重复选择同一个 teaching_task
5. 不能重复选择同一门 course 的同一学期课程
6. 退课时成绩不能已录入
7. 成绩必须在 0 到 100
8. 教师只能录入自己任课课程的成绩
```

验收标准：

```text
1. 学生能选课、退课、查成绩
2. 教师能录入自己课程的成绩
3. 选课和退课后 selected_count 正确变化
4. 事务失败时不会出现选课记录和人数不一致
```

### 第 8 阶段：统计报表与图形接口

统计报表是任务书明确评分项，不应放弃。前端可以后做，但后端数据接口必须先准备。


```text

```

```text
优秀：90-100
良好：80-89
中等：70-79
及格：60-69
不及格：0-59
```

验收标准：

```text
1. 每个统计接口能返回可直接给 ECharts 使用的数据
2. 统计结果来自真实数据库，不写死模拟结果
3. 至少覆盖学院人数、专业人数、课程选课人数、成绩分布四类图表
```
