## 推荐章节结构

### 第 1 章 课程设计任务与项目概述

写清楚：

* 题目：教务管理系统。
* 开发背景：学校教务管理涉及学院、专业、班级、学生、教师、课程、选课、成绩等数据，人工管理效率低、易出错。
* 设计目标：实现一个基于 B/S 架构的教务管理系统。
* 技术路线：Spring Boot、Maven、MySQL、JdbcTemplate、Vue 3、Element Plus、ECharts。
* 项目完成范围：基础数据管理、权限登录、任课管理、学生选课、成绩录入、统计分析、图形报表。

这一章不要太长，主要是把“为什么做、做什么、用什么做”说清楚。

### 第 4 章 数据库设计

这一章必须详细写，是课程评分重点。

任务书要求从业务中提取实体、实体联系，并完成数据库设计。 你的数据库设计建议围绕 **11 张核心业务表 + 1 张扩展日志表** 展开：`sys_user`、`college`、`major`、`department`、`class_info`、`student`、`teacher`、`course`、`teaching_task`、`student_course`、`teacher_student`、`operation_log`。

建议小节：

**4.1 实体关系分析**

写：

```text
college → major → class_info → student
college → department → teacher
teacher + course → teaching_task
student + teaching_task → student_course
teacher + student → teacher_student
```

**4.2 E-R 图设计**

放 E-R 图。这个图很重要，建议单独占一页。

**4.3 数据表结构设计**

每张表用表格写字段：

| 字段名 | 类型 | 主键/外键 | 说明 |
| --- | -- | ----- | -- |

不要把所有 SQL 贴上去。报告里写表结构即可，完整 SQL 可以作为附录或项目文件说明。

**4.4 数据库设计原则**

写几个关键点：

* 主键统一使用 `id`。
* 学号、工号、课程号等作为唯一业务编号。
* 成绩放在 `student_course`，不是放在 `student` 或 `course`。
* 教师与课程通过 `teaching_task` 建立多对多关系。
* 统计报表通过 SQL 聚合实时查询，不单独建统计表。

这里容易拿分，因为它体现你理解业务，而不是只会建表。

### 第 9 章 系统特色、不足与改进

这一章不要写空话。

**9.1 系统特色**

可以写：

* 使用前后端分离结构。
* 使用角色权限控制。
* 数据库设计覆盖教务核心业务关系。
* 支持统计报表和 ECharts 图形展示。
* 使用分层架构提高可维护性。

**9.2 当前不足**

要真实：

* 权限控制仍属于基础级别，未使用完整 RBAC。
* 密码安全可以进一步加强，如 BCrypt 加密。
* 前端页面仍以基础管理为主，交互体验可以继续优化。
* 部署主要面向本地运行，尚未做服务器完整发布。
* 复杂教务规则，如时间冲突检测、绩点计算、补考重修，仍可扩展。

**9.3 后续改进方向**

写：

* 增加课程时间冲突检测。
* 增加成绩绩点计算。
* 增加导入导出 Excel。
* 增加操作日志审计。
* 增加更细粒度权限管理。
