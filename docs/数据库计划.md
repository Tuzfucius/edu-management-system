按任务书，数据库必须体现：学院、专业、教研室、教师、班级、学生、课程、学生选课成绩、教师任课时间、教师指导学生，以及权限管理和统计报表支撑。
我建议设计成 **11 张核心业务表 + 1 张扩展日志表**。不要把所有关系硬塞进学生表、课程表，否则后面选课、成绩、统计会很难写。

# 一、总表结构

```text
权限用户类：
1. sys_user              用户登录表

组织结构类：
2. college               学院表
3. major                 专业表
4. department            教研室表
5. class_info            班级表

人员类：
6. student               学生表
7. teacher               教师表

课程业务类：
8. course                课程基础信息表
9. teaching_task         任课/开课安排表
10. student_course       学生选课与成绩表
11. teacher_student      教师指导学生表

扩展类：
12. operation_log        操作日志表，可选但建议保留
```

核心关系如下：

```text
college 1 ── N major 1 ── N class_info 1 ── N student

college 1 ── N department 1 ── N teacher

teacher N ── N course
通过 teaching_task 实现

student N ── N teaching_task
通过 student_course 实现，成绩也放在这里

teacher N ── N student
通过 teacher_student 实现
```

# 二、设计原则

## 1. 主键统一用 `id`

所有表统一：

```sql
id BIGINT PRIMARY KEY AUTO_INCREMENT
```

这样后端写 DAO、Service、查询时比较统一。

## 2. 编号字段单独设置唯一约束

例如：

```text
student_no 学号
teacher_no 工号
course_code 课程号
college_code 学院编号
```

这些是业务编号，不能当数据库主键。数据库主键用 `id`，业务编号用 `UNIQUE`。

## 3. 不建议真删除数据

建议所有主要表都有：

```text
status TINYINT DEFAULT 1
```

含义：

```text
1 正常
0 停用 / 删除
```

原因：
学生有成绩、课程有选课、教师有任课记录时，直接删除会破坏历史数据。

## 4. 报表不单独建表

统计报表通过 SQL 聚合实时查：

```text
学院学生人数
课程选课人数
成绩分布
教师任课数量
```

这些不需要单独存成表。

这是整个数据库的关键表。它表示：


| 字段名            |           类型 | 约束                        | 说明               |
| -------------- | -----------: | ------------------------- | ---------------- |
| id             |       BIGINT | PK                        | 任课记录ID           |

## 10. `student_course` 学生选课与成绩表

这张表表示学生选课，也是成绩表。

| 字段名              |           类型 | 约束                        | 说明        |
| ---------------- | -----------: | ------------------------- | --------- |
| id               |       BIGINT | PK                        | 选课记录ID    |
| student_id       |       BIGINT | FK                        | 学生ID      |
| teaching_task_id |       BIGINT | FK                        | 任课记录ID    |
| select_time      |     DATETIME | DEFAULT CURRENT_TIMESTAMP | 选课时间      |
| score            | DECIMAL(5,2) | NULL                      | 成绩，0-100  |
| grade_status     |      TINYINT | DEFAULT 0                 | 0未录入，1已录入 |
| status           |      TINYINT | DEFAULT 1                 | 1已选，0已退课  |
| remark           | VARCHAR(255) | NULL                      | 备注        |
| created_at       |     DATETIME | DEFAULT CURRENT_TIMESTAMP | 创建时间      |
| updated_at       |     DATETIME | DEFAULT CURRENT_TIMESTAMP | 更新时间      |

外键：

```sql
student_id REFERENCES student(id)
teaching_task_id REFERENCES teaching_task(id)
```

唯一约束：

```sql
UNIQUE(student_id, teaching_task_id)
```

重要说明：
成绩必须放在这张表，而不是单独放 `student` 或 `course` 表。因为成绩属于：

```text

```

不是学生本身属性，也不是课程本身属性。

# 五、为什么不这样设计

## 1. 不把教师ID放进 `course`

错误设计：

```text
course(id, course_name, teacher_id)
```

问题：
一门课可能由多个教师教，教师也可能教多门课。这样会破坏多对多关系。

正确做法：

```text
course
teacher
teaching_task
```

---

## 2. 不单独设计 `grade` 表

错误设计：

```text
grade(id, student_id, course_id, score)
```

问题：
它不知道这个成绩属于哪一次开课、哪个教师、哪个学期。

正确做法：

```text
student_course(student_id, teaching_task_id, score)
```

---

## 3. 学生表不直接存 `major_id`

错误设计：

```text
student(id, class_id, major_id)
```

问题：
`class_id` 已经能推出专业，重复存 `major_id` 会导致数据不一致。

正确做法：

```text
student.class_id → class_info.major_id → major.id
```
