# 教务管理系统项目说明文档

本文档按课程设计报告口径整理，结合当前仓库的实际技术栈与目录结构，对系统目标、总体方案、核心模块、程序结构、数据设计、测试思路和总结进行统一说明。

## 1. 课程设计任务书

### 1.1 选题背景
高校教务管理涉及学院、专业、班级、教师、学生、课程、任课任务、选课、成绩和统计报表等多个环节，数据关系复杂，人工处理容易出现重复录入、信息不一致、查询效率低和权限边界不清等问题。

本项目围绕“教务管理系统”展开，目标是通过 B/S 架构和前后端分离方式，将常见教务业务统一到一个可维护、可扩展的 Web 系统中，提升教务信息处理效率和业务规范性。

### 1.2 设计任务
系统需要完成以下基础任务：

1. 提供登录、角色识别与访问控制能力。
2. 完成学院、专业、班级、教研室、学生、教师、课程和用户账号等基础信息管理。
3. 支持教师任课任务分配、学生自主选课与退课、成绩录入与查询。
4. 提供统计报表和基础可视化能力，辅助管理决策。
5. 记录关键操作日志，保留必要的审计信息。
6. 对异常输入、权限越界和业务冲突给出统一提示。

### 1.3 系统核心建设目标
系统建设重点不在于单一页面展示，而在于形成完整业务闭环。核心目标包括：

- 数据一致：确保基础信息、选课关系和成绩记录之间保持约束关系。
- 流程完整：覆盖“登录 - 维护基础信息 - 安排任课 - 学生选课 - 成绩录入 - 报表统计”的主流程。
- 权限清晰：不同角色仅能访问其职责范围内的功能。
- 易于维护：采用分层结构拆分控制层、业务层和数据访问层。
- 便于扩展：后续可继续补充排课、补考、重修、消息通知等能力。

### 1.4 开发环境与技术路线
当前项目采用以下技术路线：

- 后端：Spring Boot 4.1.0、Java 17、Spring Web MVC
- 数据访问：Spring JDBC / JdbcTemplate
- 数据库：MySQL
- 前端：Vue 3、Vite、Element Plus、Vue Router、Axios、ECharts
- 工程管理：Maven
- 开发环境：Windows 11

技术实现上，后端负责统一接口、业务校验、事务处理和异常封装；前端负责页面交互、路由跳转、表单校验与图表展示；数据库负责持久化教务核心实体与业务关系。

## 2. 方案设计与功能说明

### 2.1 总体方案设计

#### 2.1.1 B/S 前后端分离架构方案
系统采用浏览器/服务器模式，前端以单页应用方式承载交互逻辑，后端以 REST 风格接口提供数据服务。页面只负责展示和事件响应，业务规则集中在服务端执行，便于统一控制权限、校验规则和异常处理。

#### 2.1.2 MVC 与分层开发设计
后端按 MVC 思想和分层开发方式组织代码：

- `controller`：接收请求、组织参数、返回结果。
- `service`：处理业务流程、权限判断和事务边界。
- `dao`：通过 `JdbcTemplate` 访问数据库。
- `entity` / `dto` / `vo`：承载实体数据、传输数据和展示数据。
- `common` / `exception` / `config`：提供统一结果、分页、配置和异常处理能力。

这种拆分方式能够降低耦合度，也便于后续测试和扩展。

#### 2.1.3 数据库访问方案（Spring JDBC / JdbcTemplate）
系统未采用 ORM 自动映射方案，而是使用 `JdbcTemplate` 直接编写 SQL 完成增删改查。其优点是 SQL 可控、执行路径清晰、便于理解数据关系，适合课程设计中展示数据库建模与 SQL 设计能力。

对于查询类接口，DAO 层负责组装结果集；对于写入类接口，Service 层负责先做业务判断，再调用 DAO 执行数据库操作，尽量减少脏数据和重复写入。

#### 2.1.4 前后端接口通信与统一返回格式设计
前后端通过 JSON 进行数据交换。接口返回统一采用结果封装对象，典型结构包括：

- 是否成功
- 提示信息
- 业务数据
- 分页信息

这样前端可以统一处理成功、失败和异常场景，减少每个页面重复写判断逻辑。

#### 2.1.5 权限控制与异常处理总体方案
系统的权限控制以“登录态 + 角色判定 + 接口拦截”为主。登录成功后保存用户身份信息，请求进入后端时由拦截器或控制器前置逻辑校验当前角色是否有权访问目标资源。

异常处理采用统一异常体系：业务规则冲突抛出业务异常，系统异常由全局异常处理器兜底，前端再通过拦截器和消息组件统一提示用户。

### 2.2 教务核心功能模块设计

#### 2.2.1 登录与角色权限控制模块
该模块负责系统入口、身份认证和权限识别。管理员、教师、学生三类角色进入系统后，看到的菜单和可操作范围不同。登录后应维持会话状态，并对敏感接口做权限限制。

#### 2.2.2 基础信息管理模块
基础信息管理覆盖学院、专业、班级、教研室、学生、教师、课程和用户账号，是整个系统的数据底座。该模块的主要要求是：

- 数据可新增、修改、删除和查询。
- 上下级组织关系明确，例如学院与专业、专业与班级之间存在层级关系。
- 删除前需要检查关联数据，避免破坏业务完整性。

#### 2.2.3 任课与开课任务管理模块
该模块用于描述“谁教什么课、在哪个学期、面向哪些学生群体”的业务关系。系统通过任课任务把教师、课程和教学安排连接起来，为后续选课、成绩录入和统计分析提供依据。

#### 2.2.4 学生自主选课与退课模块
学生可在开放时间范围内查看可选课程、确认容量、提交选课或退课申请。该模块需要重点处理以下约束：

- 课程容量是否已满。
- 学生是否已经选过同一课程。
- 当前选课是否处于允许退选的业务阶段。

#### 2.2.5 成绩录入与学分绩点折算模块
教师可按权限录入学生课程成绩，系统应检查成绩范围是否合法，并根据课程学分和成绩区间折算绩点。该模块后续还可扩展补考、重修、缓考等复杂规则。

#### 2.2.6 数据可视化与统计报表模块
统计模块面向管理视角，通常包括：

- 学院学生数量统计
- 课程选课人数统计
- 成绩分布统计
- 教师授课统计
- 业务日志概览

前端使用 ECharts 将后端汇总结果展示为柱状图、折线图、饼图等形式，便于快速查看数据趋势。

#### 2.2.7 系统日志与操作追踪模块
关键业务操作应记录操作人、操作时间、操作类型、请求路径和结果状态，便于后续审计和问题排查。日志模块重点记录登录、基础信息变更、选课、退课、成绩录入等敏感操作。

#### 2.2.8 系统异常处理模块
系统异常处理覆盖登录失败、权限不足、参数错误、重复数据、容量超限、成绩越界和级联删除冲突等场景。目标是让系统在出错时给出明确、可理解的提示，而不是直接暴露底层堆栈。

## 3. 程序框图（或类图，或算法原理）

### 3.1 系统全局包结构与 MVC 调用关系框图

```text
前端页面 / 请求
        |
        v
Controller  ->  参数接收、接口编排、结果返回
        |
        v
Service     ->  业务规则、权限判断、事务控制
        |
        v
Dao         ->  JdbcTemplate 执行 SQL
        |
        v
MySQL       ->  核心业务表、日志表
```

当前仓库的后端包结构已经按这一思路拆分为 `controller`、`service`、`dao`、`entity`、`dto`、`common`、`config` 和 `exception` 等目录，便于分别维护接口层、业务层和基础设施层。

### 3.2 系统核心实体类图（UML Class Diagram）
系统的核心实体关系可以概括为：

- `College` 关联多个 `Major`
- `Major` 关联多个 `ClassInfo`
- `ClassInfo` 关联多个 `Student`
- `College` 关联多个 `Department`
- `Department` 关联多个 `Teacher`
- `Course` 通过 `TeachingTask` 与教师和学期安排关联
- `Student` 通过 `StudentCourse` 与课程形成选课关系
- `Teacher` 通过 `TeacherStudent` 与学生形成指导或管理关系

这些实体关系共同支撑教务系统的基础管理、教学安排和成绩业务。

### 3.3 登录与角色权限拦截流程图

```text
用户输入账号密码
        |
        v
后端校验账号是否存在
        |
        +---- 否 -> 返回登录失败
        |
        v
校验密码与角色信息
        |
        +---- 否 -> 返回身份校验失败
        |
        v
写入登录态 / 会话信息
        |
        v
请求进入接口时检查角色权限
        |
        +---- 无权限 -> 返回 403 风格提示
        |
        v
允许访问目标功能
```

该流程强调“先认证、再授权”的原则，避免未登录用户或越权角色直接访问业务接口。

### 3.4 学生在线选课容量判定流程图

```text
学生发起选课请求
        |
        v
检查当前课程是否存在
        |
        v
检查学生是否已经选过该课程
        |
        +---- 是 -> 返回“请勿重复选课”
        |
        v
检查课程当前已选人数是否达到容量上限
        |
        +---- 是 -> 返回“课程容量已满”
        |
        v
写入选课关系表
        |
        v
返回选课成功
```

该流程的关键在于先做业务约束判断，再执行落库，尽量降低并发冲突和脏数据风险。

### 3.5 教师成绩录入合规校验与 GPA 折算逻辑
成绩录入时通常按以下逻辑执行：

1. 检查当前教师是否对该课程有录入权限。
2. 检查学生是否属于该教学任务对应的选课名单。
3. 校验成绩是否在合法范围内，一般为 `0~100`。
4. 按预设规则折算绩点，并结合课程学分计算加权 GPA。

若后续需要支持补考、重修或等级制成绩，可在这一层继续扩展映射规则。

### 3.6 统一异常处理流程图

```text
业务层 / 数据层抛出异常
        |
        v
全局异常处理器捕获
        |
        +---- 业务异常 -> 提示具体业务原因
        |
        +---- 参数异常 -> 提示参数格式或范围问题
        |
        +---- 系统异常 -> 记录日志并返回通用错误信息
        |
        v
前端统一消息提示
```

统一异常处理的目标是让接口返回稳定、可预测，同时方便前端做统一提示和日志排查。

## 4. 主要程序清单与数据结构设计

### 4.1 主要程序清单（类结构与接口定义）

#### 4.1.1 身份认证控制器与安全拦截器
设计上由以下组件完成：

- `AuthController`
- `AuthService`
- `AuthInterceptor`
- `WebConfig`

其职责是完成登录、退出、身份识别、会话控制和接口拦截。当前仓库中已存在 `AuthController`，其余组件可按后续实现继续补齐。

#### 4.1.2 学生与教师信息服务类
设计上可由以下组件承担：

- `StudentController`
- `StudentService`
- `StudentDao`
- `TeacherService`

主要负责学生、教师基础档案的查询、维护和关联检查。当前仓库中已存在学生、教师控制器，后续可结合统一的数据访问方式补充服务层实现。

#### 4.1.3 课程、任课与选课业务逻辑类
该部分建议由以下组件组成：

- `CourseService`
- `TeachingTaskService`
- `SelectionService`

它们负责课程维护、任课任务配置、学生选课与退课、容量判断和业务冲突检查。

#### 4.1.4 成绩录入与 GPA 折算业务类
该部分建议由以下组件组成：

- `GradeController`
- `GradeService`
- `GradeDao`

职责包括成绩录入、成绩修改、成绩查询、GPA 计算和成绩合法性校验。

#### 4.1.5 报表统计与 ECharts 数据桥接类
该部分建议由以下组件组成：

- `ReportController`
- `ReportService`
- `ChartDataVO`

其作用是把数据库聚合结果整理成前端图表需要的结构，减少页面端的数据转换工作。

#### 4.1.6 统一返回、分页与异常处理类
当前项目中已具备或规划具备的通用类包括：

- `Result`
- `PageResult`
- `BusinessException`
- `GlobalExceptionHandler`

另外，`RequestLoggingFilter` 可用于记录请求轨迹和接口访问情况，辅助排查问题。

### 4.2 数据结构设计

#### 4.2.1 逻辑结构设计
逻辑上，系统围绕“组织结构 - 人员信息 - 教学业务 - 操作记录”四类对象展开。组织结构用于描述学院、专业、班级和教研室；人员信息用于描述学生、教师和账号；教学业务用于描述课程、任课、选课和成绩；操作记录用于描述系统行为轨迹。

#### 4.2.2 物理结构设计
当前课程设计口径下，可按 12 张 MySQL 业务物理表进行描述：

- `sys_user`
- `college`
- `major`
- `department`
- `class_info`
- `student`
- `teacher`
- `course`
- `teaching_task`
- `student_course`
- `teacher_student`
- `operation_log`

其中前 11 张表承载核心业务，第 12 张表用于日志和扩展追踪。

#### 4.2.3 数据一致性与约束设计
数据层应重点设计以下约束：

- 唯一约束：账号、编号、课程代码等字段避免重复。
- 外键或逻辑关联：保证专业属于学院、班级属于专业等关系成立。
- 状态字段：区分启用、停用、已删除、已结束等业务状态。
- 删除限制：存在下游关联数据时不允许直接物理删除。

这些约束可减少人工误操作带来的数据污染。

#### 4.2.4 事务处理设计
涉及多表联动的业务应纳入事务控制，重点包括：

- 选课与退课
- 成绩录入与成绩修改
- 用户与学生信息联动

事务的目标是保证“要么全部成功，要么全部回滚”，避免数据只改了一半。

### 4.3 系统出错处理设计

#### 4.3.1 出错信息分类
系统常见错误可分为：

- 登录异常
- 权限异常
- 参数异常
- 重复数据
- 容量超限
- 成绩越界
- 级联删除冲突

#### 4.3.2 后端异常响应机制
后端建议采用“业务异常 + 全局异常处理器 + 统一 Result 返回”的方式。业务上可预期的问题抛出 `BusinessException`，系统未预料到的问题交给全局处理器统一兜底。

#### 4.3.3 前端异常提示机制
前端可通过 Axios 响应拦截器统一识别错误状态，并使用 Element Plus 的消息组件给出提示。这样可以避免每个页面重复编写错误处理逻辑。

#### 4.3.4 典型异常处理流程
典型场景包括：

- 重复选课：提示学生已选过该课程。
- 容量已满：提示课程名额不足。
- 成绩越界：提示成绩必须在合法范围内。
- 越权访问：提示当前角色无权限操作。

## 5. 程序运行测试与分析

### 5.1 测试程序与数据设计

#### 5.1.1 测试环境配置
测试环境应覆盖后端服务、MySQL 数据库和前端页面联调三个层面，确保接口、页面和数据库三端一致。

#### 5.1.2 数据库初始化与模拟数据设计
初始化数据建议包含学院、专业、班级、教师、学生、课程和任课任务等基础样本，便于快速验证核心流程是否可跑通。

#### 5.1.3 功能测试用例设计
测试用例应覆盖主业务流程：

- 登录与退出
- 基础信息增删改查
- 任课任务维护
- 学生选课与退课
- 成绩录入与查询
- 统计图表展示

#### 5.1.4 边界值与异常场景测试设计
需要重点验证：

- 空参数和非法参数
- 重复提交
- 课程满员
- 角色越权
- 分数上下界

#### 5.1.5 接口测试设计
接口测试可通过 Apifox、浏览器直接请求和前端联调三种方式完成。接口设计应保证请求参数、返回结构和错误状态统一。

### 5.2 运行结果与分析

#### 5.2.1 系统启动与数据库连接测试结果
系统启动后应首先确认数据库连接、连接池配置和基础表读写是否正常，保证后续业务接口能够稳定执行。

#### 5.2.2 登录与角色权限测试结果
登录测试应确认三类角色能够进入各自页面，未登录状态和越权访问应被拦截。

#### 5.2.3 基础信息管理测试结果
基础信息模块应能够完成新增、修改、查询和删除，同时对有依赖关系的数据给出合理限制。

#### 5.2.4 任课任务与学生选课测试结果
任课任务发布后，学生端能够按条件查看课程并执行选课或退课，容量超限和重复选课应被正确拦截。

#### 5.2.5 成绩录入与 GPA 查询测试结果
成绩录入后应能够在学生端或管理端查询结果，并按既定规则生成 GPA 或统计信息。

#### 5.2.6 统计报表与图形分析测试结果
统计模块应能够把汇总数据转换成图表数据，图表展示应与数据库统计结果保持一致。

#### 5.2.7 异常处理结果分析
异常处理测试的重点是：错误信息是否清晰、前端提示是否一致、后端是否能够稳定返回统一结构。

#### 5.2.8 核心业务闭环验证分析
如果登录、基础信息、任课、选课、成绩和报表都能顺利串联，则说明系统已经形成较完整的业务闭环，能够满足课程设计的基本要求。

## 6. 结论与心得

### 6.1 优点

#### 6.1.1 前后端分离结构清晰
前端与后端职责明确，接口边界清楚，便于并行开发和后续维护。

#### 6.1.2 后端分层设计降低耦合
控制层、业务层和数据访问层分离后，代码更容易测试和替换。

#### 6.1.3 数据库关系设计完整
系统围绕组织结构和教学业务建立数据关系，能够支撑主要教务流程。

#### 6.1.4 多角色权限控制明确
管理员、教师和学生的功能范围区分清楚，符合实际教务使用场景。

#### 6.1.5 选课、成绩、报表形成业务闭环
系统不是单点功能，而是从选课到成绩再到统计报表形成了完整链路。

#### 6.1.6 异常处理与数据校验较完整
通过统一异常和业务校验，可以减少用户误操作带来的问题。

### 6.2 不足

#### 6.2.1 权限控制尚未扩展为完整 RBAC 模型
当前权限粒度适合课程设计，但若面向真实业务，仍需更细的权限配置能力。

#### 6.2.2 密码安全性仍可进一步增强
密码加密、盐值策略、登录失败限制等机制还可以继续完善。

#### 6.2.3 选课高并发控制尚未引入 Redis 或分布式锁
当选课人数很多时，还需要更强的并发控制方案。

#### 6.2.4 排课冲突检测算法仍较简化
复杂排课场景下，还需要进一步考虑时间冲突、教室冲突和教师冲突。

#### 6.2.5 GPA 规则尚未覆盖补考、重修等复杂教务场景
当前规则更适合基础课程设计，真实业务中还需要更完整的学分绩点体系。

#### 6.2.6 系统尚未进行服务器部署和压力测试
当前更侧重功能实现与结构展示，运行稳定性和性能数据还有待补充。

### 6.3 收获与体会

#### 6.3.1 对 Java Web 全栈开发流程的理解
通过本项目可以完整接触需求、建模、编码、联调和测试等环节。

#### 6.3.2 对数据库逻辑结构与物理结构设计的理解
系统设计过程中可以明显感受到逻辑建模和物理落库之间的差异。

#### 6.3.3 对 Controller / Service / Dao 分层思想的理解
分层不是形式，而是为了把职责拆清楚，减少后续维护成本。

#### 6.3.4 对权限控制、异常处理与接口联调的实践体会
这些基础能力虽然不显眼，但决定了系统是否稳定、是否可用。

#### 6.3.5 对复杂业务闭环设计能力的提升
从单点 CRUD 到完整业务链路的设计，是本项目最有价值的训练点。
