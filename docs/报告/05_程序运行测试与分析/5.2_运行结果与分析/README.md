# 5.2 运行结果与分析

本节根据系统启动、登录权限、基础信息、任课选课、成绩 GPA、报表图形、异常处理和业务闭环等方面分析运行结果。结果分析以测试现象和业务意义为主，不展示测试源代码。

## 5.2.1 系统启动与数据库连接测试结果

系统启动测试首先验证数据库服务、后端应用和前端应用是否能够依次启动。数据库初始化完成后，后端应用读取 `application.yaml` 中的连接配置，通过 MySQL 驱动连接 `edu_management_system`。数据库连接测试通过后，说明数据库地址、用户名、密码、驱动类和字符集配置基本正确。后端应用启动在 8080 端口，并带有 `/edu` 上下文路径；前端开发服务启动后，浏览器能够访问登录页面，并通过代理向后端发送接口请求。日志文件能够记录运行信息，说明日志配置也处于可用状态。

从运行结果看，系统具备本地课程设计演示所需的基本环境闭环。数据库脚本能够建立完整表结构，模拟数据能够支撑页面显示；后端接口能够访问数据库并返回统一结果；前端页面能够渲染登录、列表、表单和图表。若数据库未启动或脚本未执行，登录接口和列表接口会失败；若后端未启动，前端会出现网络请求错误；若代理配置错误，前端路径和后端上下文会不匹配。因此系统启动测试不仅证明单个程序可运行，也证明前后端分离项目的多组件协作可行。该结果表明系统已经具备继续进行功能测试和业务流程测试的基础。

## 5.2.2 登录与角色权限测试结果

登录测试分别使用管理员、教师和学生账号进行验证。管理员账号登录后应进入管理员工作台，能够访问基础信息管理、课程管理、报表统计和操作日志等页面；教师账号登录后应进入教师工作台，能够查看本人任课课程、学生名单和成绩录入页面；学生账号登录后应进入学生工作台，能够查看可选课程、已选课程和个人成绩。错误密码、错误角色或未登录访问受保护页面时，系统应返回失败提示或跳转登录、无权限页面。测试结果表明，系统能够根据 `sys_user` 中的角色字段完成三类用户的基本分流。

从分析角度看，当前权限控制满足课程设计中的角色区分要求，但仍属于轻量实现。前端路由守卫和菜单控制可以有效减少普通用户误入其他角色页面的情况，登录接口返回的用户信息也能支持页面展示和后续业务查询。然而，完整生产系统还需要更严格的后端统一权限拦截，防止绕过前端直接调用接口。当前报告应客观评价：系统已经实现管理员、教师、学生三角色登录和页面级访问控制，能够满足课程设计演示；但尚未形成细粒度 RBAC 权限模型，也未引入 Spring Security、JWT 或加密密码机制。该分析既肯定了现有功能，也说明了后续安全增强方向。

## 5.2.3 基础信息管理测试结果

基础信息管理测试覆盖学院、专业、教研室、班级、课程、用户、学生和教师等模块。测试通过列表查询验证初始化数据能够正常展示，通过新增和修改验证表单输入能够写入数据库并刷新页面，通过停用或删除验证状态字段和关联限制是否生效。学院、专业、教研室、班级和课程等基础模块具有较清晰的分层结构，控制器调用服务层，服务层调用数据访问层，数据访问层通过 JdbcTemplate 操作数据库。学生和教师维护还涉及账号绑定、组织归属和业务编号唯一性，因此比单纯字典表更能体现数据关系。

运行结果表明，基础信息模块能够为后续业务提供稳定主数据。新增专业时必须选择学院，新增班级时必须选择专业，新增学生时必须绑定班级和用户，新增教师时必须绑定教研室和用户；这些约束保证了后续任课、选课和统计查询能够沿着关系链追溯。重复编号或被下游记录引用的数据操作会受到限制，说明唯一约束和关联校验发挥了作用。基础信息测试的意义在于验证数据库设计是否可用，因为如果基础数据出现矛盾，例如学生班级不存在或课程编号重复，后续业务流程必然受到影响。测试结果说明系统的基础数据层能够支撑教务核心流程。

## 5.2.4 任课任务与学生选课测试结果

任课任务测试验证管理员能否把课程、教师、学期、上课时间、教室和容量组合成开课任务。创建成功后，教师端能够按教师身份查看本人任课，学生端能够按学期查看可选任务。学生选课测试验证学生点击选课后，系统能够新增选课记录并同步增加任课任务已选人数；退课后，选课记录状态改变，已选人数同步减少。重复选课、容量已满和时间冲突等异常场景应被拒绝。测试结果表明，系统通过 `teaching_task` 和 `student_course` 两张表形成了课程从开设到被学生选择的业务链路。

该模块的分析重点是数据一致性。任课任务的容量和已选人数直接影响学生端可选状态，选课记录又影响教师端学生名单和成绩录入。因此选课成功必须同时更新选课表和任课任务人数，退课也必须同步回退人数。当前项目在课程设计场景下能够完成这一闭环，并通过唯一约束和业务校验防止重复选课。需要客观说明的是，高并发抢课场景尚未引入 Redis、消息队列、分布式锁或数据库条件更新等加强机制，因此更适合本地演示和低并发教学环境。整体来看，该模块已经实现了教务系统最关键的业务流程之一，即课程资源被学生实际选择并进入成绩管理阶段。

## 5.2.5 成绩录入与 GPA 查询测试结果

成绩录入测试以教师角色进行。教师进入成绩管理页面后，系统根据教师身份查询本人任课下的学生选课记录。教师输入 0 到 100 范围内的成绩并提交后，后端更新 `student_course` 表中的成绩字段和成绩状态。学生端进入我的成绩页面后，可以查看课程名称、学期、教师、学分、成绩和成绩状态，并根据已录入成绩统计平均分、已获学分和待出成绩数量。越界成绩如 -1 或 101 应被拒绝，未录入成绩不应参与平均分和绩点统计。

当前项目实际实现以百分制成绩、平均分、及格学分和成绩状态展示为主，没有独立落库的 GPA 字段，也没有单独的 GPA 后端接口。因此报告中的 GPA 查询应表述为“基于成绩和学分的绩点折算设计”，而不是夸大为已经完整实现复杂 GPA 体系。课程设计可以按百分制分段规则说明 GPA 扩展逻辑：先把单科成绩折算为绩点，再按学分加权计算综合 GPA。测试结果说明成绩录入链路已经打通，即教师端可以录入，数据库可以保存，学生端可以查询，报表端可以统计成绩分布。后续如需更贴近真实教务系统，可继续补充补考、重修、缓考、免修和等级制课程的成绩规则。

## 5.2.6 统计报表与图形分析测试结果

统计报表测试包括概览指标、学院学生人数、成绩分布和教师任课负载。后端报表接口从数据库聚合查询结果，前端管理员报表页面通过 ECharts 展示图形，并通过统计卡片展示关键数量。学院学生人数能够反映不同学院的学生规模，成绩分布能够反映已录入成绩在各分数区间的分布情况，教师任课负载能够反映教师承担课程任务的数量或规模。教师端也可以基于本人任课下的学生成绩数据查看已录成绩数、平均分、及格率和成绩区间。

运行结果表明，系统不只是完成了数据录入，还能将业务数据转化为管理分析结果。报表接口不单独依赖统计表，而是直接基于业务表聚合，避免了统计数据与业务数据不同步的问题。前端 ECharts 图表提升了数据可读性，使管理员可以快速判断学院规模、课程成绩情况和教师工作量。需要注意的是，部分图表展示数据在前端页面中存在二次计算，因此后续若报表口径复杂化，应优先把统计规则收敛到后端服务层，前端只负责展示。整体而言，报表模块形成了“业务数据产生、数据库保存、接口聚合、图表展示”的完整反馈链。

## 5.2.7 异常处理结果分析

异常处理测试覆盖登录失败、参数缺失、重复数据、关联删除冲突、重复选课、容量已满、成绩越界和未登录访问等场景。测试结果应表现为：系统返回统一结构的失败响应，前端显示可理解的消息提示，数据库状态不被错误操作破坏。例如重复学号新增失败后，学生表不应新增记录；课程容量已满时，选课表和已选人数不应变化；成绩 101 分提交失败后，原成绩不应被覆盖；未登录访问当前用户接口时，应提示未登录或跳转登录页面。

从分析角度看，异常处理模块提高了系统健壮性。教务系统中的错误往往具有业务含义，不宜简单显示“服务器错误”。当前系统通过业务校验、数据库约束、全局异常处理和前端消息提示构成多层防护。数据库约束保证底线，服务层校验提供业务语义，全局异常处理统一响应格式，前端提示负责用户交互。需要说明的是，当前业务异常和运行时异常的使用仍可进一步规范，部分场景可以更细致地区分错误码和 HTTP 状态。总体来看，异常处理结果证明系统能够应对常见错误输入，满足课程设计对出错处理设计的要求。

从数据安全角度看，异常路径的测试还证明了系统具备“失败即不改变核心数据”的基本原则。对于教务系统而言，这一点比提示是否美观更重要，因为错误写入可能影响学生选课资格、教师成绩统计和管理员报表口径。

## 5.2.8 核心业务闭环验证分析

核心业务闭环验证从管理员初始化基础数据开始，依次经过创建课程、创建任课任务、学生选课、教师录入成绩、学生查看成绩、管理员查看报表和日志追踪。该闭环覆盖系统最重要的实体关系：学院、专业、班级支撑学生归属，教研室支撑教师归属，课程与教师通过任课任务形成教学安排，学生与任课任务通过选课成绩表形成学习记录，成绩和选课记录进一步进入报表统计。若闭环中任何一个环节失败，后续功能都会受影响。

运行结果表明，当前系统已经具备教务管理系统的基本业务完整性。管理员可以维护主数据，教师可以管理本人课程成绩，学生可以参与选课并查看结果，报表可以基于业务数据提供统计反馈，日志可以记录关键操作。这种闭环比单纯 CRUD 更能体现课程设计价值，因为它将数据库设计、后端分层、前端交互、接口通信、异常处理和统计分析连接成一个整体。系统仍有可提升空间，例如更严格权限控制、更安全密码存储、更复杂排课算法和高并发选课控制，但作为 Java Web 课程设计，已经能够较完整地展示从需求到实现、从数据到界面、从正常路径到异常路径的开发过程。
