# 5.1 测试程序与数据设计

本节说明测试环境、数据库初始化、模拟数据、功能测试用例、边界异常场景和接口联调方案。测试设计遵循“先验证环境连通，再验证单模块功能，最后验证业务闭环”的顺序。

## 5.1.1 测试环境配置

本系统测试环境以本地 Windows 11 为基础，后端运行环境为 Java 17、Maven、Spring Boot 4.1.0、Spring Web MVC、Spring JDBC/JdbcTemplate 和 MySQL，前端运行环境为 Vue 3、Vite、Element Plus、Vue Router、Axios 和 ECharts。后端服务端口配置为 8080，上下文路径为 `/edu`；前端开发服务通常由 Vite 启动，默认端口为 5173，并通过开发代理把 `/api` 请求转发到后端接口。数据库连接配置位于 `application.yaml`，默认连接本机 `edu_management_system` 数据库，用户名为 `root`，密码可通过环境变量或本地配置提供。日志文件写入 `logs/edu-management-system.log`，便于观察请求过程和异常信息。

测试环境配置的目标是保证前端、后端、数据库三者能够稳定联通。首先通过数据库连接测试确认 MySQL 服务可访问，数据库驱动和连接地址正确；其次启动后端应用，检查端口、上下文路径和日志输出是否正常；再次启动前端开发服务，检查登录页面、路由跳转、接口代理和静态资源是否可访问。由于系统采用前后端分离结构，单独启动前端但不启动后端会导致接口请求失败；单独启动后端但不初始化数据库会导致登录和列表查询失败。因此测试前必须明确三个前提：数据库已初始化，后端服务已启动，前端代理指向正确后端地址。只有环境连通后，后续功能测试和接口联调才具有有效性。

## 5.1.2 数据库初始化与模拟数据设计

数据库初始化按三个脚本顺序执行：先创建数据库，再创建 12 张业务表，最后插入模拟数据。核心表包括 `sys_user`、`college`、`major`、`department`、`class_info`、`student`、`teacher`、`course`、`teaching_task`、`student_course`、`teacher_student` 和 `operation_log`。模拟数据覆盖管理员、教师和学生三类账号，覆盖学院、专业、教研室、班级、课程、任课任务、选课成绩和指导关系，能够支撑登录、基础信息维护、选课、成绩录入和报表统计等测试。样例数据中包含多名学生、多位教师、多门课程和多个学期任务，使测试不仅停留在空表操作，而是能够验证关联查询和统计聚合。

模拟数据设计需要兼顾正常路径和异常路径。正常路径数据包括可登录账号、有效学生、有效教师、正常课程、可选任课任务和已录入成绩记录，用于验证页面展示和业务闭环。异常路径数据则通过测试操作构造，例如使用重复学号验证唯一约束，使用不存在的班级编号验证外键或业务校验，使用已满容量任务验证选课容量限制，使用越界成绩验证成绩范围检查。对课程设计报告而言，模拟数据的意义不只是“让页面有内容”，而是提供可重复验证的测试基础。每个测试用例应明确输入数据、操作步骤、预期结果和实际结果，例如使用管理员账号登录后应进入管理员首页，使用学生账号查询可选课程应返回带容量和教师信息的课程列表，教师录入 88 分后学生端应能查询到对应成绩。

## 5.1.3 功能测试用例设计

功能测试用例按照角色和模块划分。登录模块测试管理员、教师、学生三类账号能否正确登录，错误密码、错误角色和停用账号能否被拒绝。基础信息模块测试学院、专业、教研室、班级、课程、学生、教师、用户账号的列表查询、新增、修改和停用功能，并关注前端表格刷新和后端数据是否一致。任课任务模块测试管理员能否创建课程与教师的开课安排，教师能否查看本人课程。学生选课模块测试学生能否查看可选课程、成功选课、查看已选课程和执行退课。成绩模块测试教师能否针对本人课程录入成绩，学生能否查询成绩和统计结果。报表模块测试概览统计、学院学生人数、成绩分布和教师任课负载是否能够返回有效数据并渲染图表。

测试用例应采用“输入、步骤、预期输出、验证依据”的形式组织。例如登录测试输入账号 `admin` 和正确密码，预期返回管理员用户信息并进入管理员页面；课程新增测试输入课程编号、课程名称、学分和考核方式，预期课程列表新增记录且课程编号唯一；选课测试输入学生编号和任课任务编号，预期选课记录增加，任课任务已选人数加一；退课测试输入选课记录编号，预期记录状态变为退课，已选人数减一；成绩录入测试输入选课记录编号和 90 分，预期成绩状态变为已录入，学生端成绩列表显示该分数。通过这些用例，可以逐步证明系统各模块能够独立运行，也能在跨模块场景中形成完整业务链路。

## 5.1.4 边界值与异常场景测试设计

边界值和异常场景测试用于证明系统在错误输入或极端条件下仍能保持数据一致。登录边界包括账号为空、密码为空、角色为空、密码错误、角色不匹配和未登录访问当前用户接口。基础信息边界包括编号重复、名称为空、外键不存在、状态非法、删除已被下游数据引用的记录。课程边界包括学分小于或等于 0、课程编号重复、课程被任课任务引用后尝试停用。任课任务边界包括容量为 0、已选人数超过容量、开始节次大于结束节次、星期值超出 1 到 7、教师同一时间段冲突。选课边界包括重复选课、容量已满、不可选任务、退课不存在记录、退课后人数不能为负。成绩边界包括成绩为空、成绩小于 0、成绩大于 100、未选课学生录入成绩和教师越权录入成绩。

这些异常测试的共同标准是失败时不能破坏数据库状态。重复选课失败后，选课记录数量不应增加，任课任务已选人数不应变化；容量已满失败后，不应产生新的学生课程记录；成绩越界失败后，原成绩和成绩状态不应被更新；删除存在下游关联的数据失败后，原记录状态不应改变。测试数据需要覆盖边界点，例如成绩 0、60、100 应作为合法值，-1 和 101 应作为非法值；容量等于已选人数时应拒绝选课，容量大于已选人数时才允许选课。通过边界测试，可以证明系统不是只在理想输入下运行，而是对常见错误、特殊情况和数据约束冲突具有防护能力。

## 5.1.5 接口测试设计（Apifox / 浏览器 / 前端页面联调）

接口测试可结合 Apifox、浏览器开发者工具和前端页面联调进行。Apifox 或类似接口工具适合直接调用后端 REST 接口，验证请求参数、返回结构、错误提示和响应数据；浏览器开发者工具适合观察前端页面发出的实际请求，包括请求路径、请求体、响应内容和网络错误；前端页面联调则验证用户操作是否能完整触发接口、更新页面状态并给出正确提示。项目提供 `docs/openapi.yaml` 作为接口说明，覆盖认证、学院、专业、教研室、班级、课程、用户、学生、教师、任课任务、选课成绩、教师指导、报表和日志等接口，可作为接口测试用例编写依据。

接口测试的重点不是只看请求是否返回成功，还要检查响应格式是否符合统一 `Result` 结构，失败时是否包含可理解提示，分页接口是否包含记录列表和总数，报表接口字段是否能被前端 ECharts 消费。联调时可以先使用 Apifox 调通登录接口，获取正常响应；再在浏览器中登录系统，观察 localStorage、路由跳转和菜单显示；随后依次操作新增课程、创建任课任务、学生选课、教师录入成绩和查看报表。若页面提示失败，应通过浏览器网络面板确认是前端参数问题、后端业务校验问题还是数据库数据问题。通过接口工具和页面联调结合，可以证明系统既有接口可测性，也具备真实用户流程可用性。
