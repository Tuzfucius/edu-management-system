# 6.1 优点

本节从架构、分层、数据库、权限、业务闭环和异常校验等方面总结系统优点。优点描述以实际工程和课程设计目标为基础，不作超出当前实现范围的夸大。

## 6.1.1 前后端分离结构清晰

系统采用 Spring Boot 后端和 Vue 3 前端分离的结构，浏览器端负责页面展示、路由跳转、表单输入、消息提示和图表渲染，服务端负责业务规则、数据访问、异常处理和统一响应。前端工程单独位于 `frontend` 目录，后端工程位于 `src/main/java` 目录，数据库脚本位于 `sql` 目录，文档位于 `docs` 目录。这种组织方式使工程边界明确，便于分别开发、测试和维护。前端通过 Axios 调用后端接口，不直接访问数据库；后端以 JSON 返回业务数据，不直接控制页面细节。

前后端分离结构的优点在课程设计中表现明显。页面调整不会影响后端数据库访问，后端业务规则调整也不会要求重写页面结构；接口文档可以作为前后端协作契约，前端页面可以通过模拟数据或接口工具先行调试，后端接口也可以通过自动化测试独立验证。该结构还便于展示现代 Web 应用开发流程，例如前端使用 Element Plus 构建表格和表单，使用 ECharts 展示报表，后端使用 Controller 提供 REST 风格接口。相比传统单体页面渲染方式，本系统更能体现工程化和模块化思想，为后续部署、扩展移动端或接入其他客户端保留了空间。

## 6.1.2 后端分层设计降低耦合

后端按照 Controller、Service、Dao、Entity、DTO、Common、Exception 等职责组织。Controller 面向 HTTP 请求，Service 处理业务规则，Dao 使用 JdbcTemplate 访问数据库，Entity 或映射对象承载数据，Common 提供统一响应，Exception 负责异常处理。这种分层使每一层承担相对单一职责，降低了代码之间的直接耦合。基础信息模块如学院、专业、教研室、班级和课程采用较完整的控制器、服务、数据访问和实体结构，综合业务模块则由 `AcademicBusinessService` 与 `AcademicBusinessDao` 集中处理跨表逻辑。

分层设计的实际好处体现在维护和测试中。控制器测试可以关注接口路径、请求参数和返回结构；服务测试可以关注业务规则，例如重复编号、关联限制、容量判断和成绩范围；数据访问测试可以关注 SQL 查询和数据库字段映射。若后续需要替换数据访问方案，理论上主要影响 Dao 层；若需要增加业务校验，主要调整 Service 层；若需要修改接口路径或返回消息，主要调整 Controller 层和统一响应。虽然当前综合业务服务仍有进一步拆分空间，但整体分层思路已经清楚，有利于课程设计答辩时说明程序结构和调用关系。

## 6.1.3 数据库关系设计完整

数据库设计覆盖了教务系统中的主要实体和关系，包括系统用户、学院、专业、教研室、班级、学生、教师、课程、任课任务、学生选课成绩、教师指导学生和操作日志。组织结构上，学院到专业、专业到班级、班级到学生形成学生培养链路；学院到教研室、教研室到教师形成教师管理链路；教学业务上，教师与课程通过任课任务关联，学生与任课任务通过选课成绩关联。这样的结构能够表达真实教务系统中“一门课程可多次开设、一个教师可教多门课、一个学生可选择多门开课任务”的业务关系。

数据库关系完整还体现在约束设计上。各表统一使用自增主键，业务编号通过唯一约束保证不重复，外键约束保证上下游关系存在，检查约束限制角色、状态、性别、成绩、容量和节次范围，状态字段支持逻辑停用。成绩放在 `student_course` 表中，体现成绩依赖具体选课记录；学生不直接保存专业字段，而是通过班级追溯到专业和学院，减少冗余和不一致。报表不单独建统计表，而是基于业务表聚合生成，保证统计结果与当前业务数据一致。整体来看，数据库设计能够较好支撑课程设计要求中的基础管理、选课、成绩和统计功能。

## 6.1.4 多角色权限控制明确

系统明确区分管理员、教师和学生三类角色。管理员负责基础信息维护、课程和任课任务管理、统计报表和日志查看；教师负责查看本人任课课程、管理学生成绩和查看教师端统计；学生负责查看可选课程、执行选课退课和查询个人成绩。登录成功后，前端根据角色跳转对应工作台，并通过路由和菜单控制页面访问。后端登录接口返回角色信息，业务接口也以用户、教师或学生标识作为查询条件，支撑不同角色看到不同数据范围。

多角色设计的优点在于贴近教务管理实际场景。教务系统不是单一用户系统，不同角色对同一数据具有不同权限和关注点。管理员关注全局管理和统计，教师关注本人课程和学生成绩，学生关注个人选课和学习结果。通过角色分流，界面更简洁，用户不需要面对与自身无关的功能；业务边界也更清楚，减少误操作可能。当前权限控制尚未达到完整 RBAC 或后端强拦截级别，但已经满足课程设计对角色权限的基本要求，为后续引入权限表、接口权限码和后端拦截器提供了清晰基础。

这种角色划分也有助于测试工作组织。测试人员可以分别以管理员、教师和学生身份验证页面入口、接口返回和数据范围，从而更容易发现越权访问、菜单误显和业务数据串扰等问题。

## 6.1.5 选课、成绩、报表形成业务闭环

系统的一个重要优点是形成了从任课任务到学生选课、从选课记录到成绩录入、从成绩数据到统计报表的业务闭环。管理员先维护课程和教师，再创建任课任务；学生基于任课任务进行选课，系统记录选课状态和已选人数；教师基于本人任课任务查看学生名单并录入成绩；学生查询个人成绩，管理员查看成绩分布和教师负载等报表。这个闭环说明系统不只是完成了若干独立 CRUD 页面，而是把教务业务中的多个环节连接起来。

业务闭环的价值在于能够验证数据模型和系统流程是否真实可用。若任课任务设计不合理，学生无法选择具体课程；若选课记录缺少状态或容量字段，退课和容量统计会混乱；若成绩不与选课记录绑定，教师成绩录入和学生查询会失去上下文；若报表无法基于业务数据聚合，管理分析就无法成立。当前系统通过 `teaching_task` 和 `student_course` 两张关键表，把课程资源、教师授课、学生选择和成绩评价连接起来，再通过报表模块反馈统计结果，体现了较完整的教务业务思维。

因此，系统展示的不是零散页面，而是一条可以被实际演示和验证的教务流程。该特点使课程报告、功能截图和答辩说明都能够围绕同一条业务主线展开。

## 6.1.6 异常处理与数据校验较完整

系统在多个层面进行了异常处理和数据校验。数据库层通过唯一约束、外键约束和检查约束防止明显非法数据进入表中；业务层对必填字段、重复编号、关联关系、容量限制、时间冲突和成绩范围进行校验；全局异常处理机制将业务异常、参数异常、数据库异常和运行时异常转换为统一响应；前端通过消息提示向用户展示错误原因。这样可以让用户在操作失败时得到明确反馈，也能避免错误操作破坏数据库状态。

异常处理与数据校验的优点在测试中非常明显。重复选课不会导致选课人数异常增加，容量已满不会继续插入记录，成绩越界不会覆盖原成绩，存在关联的数据不会被轻易删除。系统不只是关注“成功时如何处理”，也考虑了“失败时如何保持一致”。对于教务管理系统而言，这一点很重要，因为学生成绩、课程容量和人员信息都具有较强的数据严肃性。虽然当前异常码和权限异常还可以进一步精细化，但已有机制已经能支撑课程设计中的边界测试和错误场景分析。

这种设计还降低了前端联调成本。前端页面能够依据统一提示展示错误原因，测试人员也能通过稳定返回结构快速判断失败属于输入问题、业务冲突还是系统异常。
