# 6.2 不足

本节客观说明系统仍存在的不足。课程设计已经完成了核心业务闭环，但在权限安全、高并发、排课算法、复杂成绩规则、部署和压力测试方面仍有继续改进空间。

## 6.2.1 权限控制尚未扩展为完整 RBAC 模型

当前系统权限控制主要基于 `sys_user` 表中的单一角色字段，角色包括管理员、教师和学生。前端根据角色控制路由和菜单，后端根据登录信息和业务参数完成部分数据范围限制。该方式适合课程设计和中小规模演示系统，结构简单，容易理解，也能满足三类角色的基本分流。但是它还不是完整 RBAC 模型，没有独立的角色表、权限表、角色权限关联表和用户角色关联表，也没有把每个接口或按钮级操作抽象为可配置权限项。

这种不足会影响系统在真实场景中的扩展能力。实际学校中可能存在教务处管理员、学院管理员、辅导员、教研室主任、任课教师、督导教师等多种角色，同一用户也可能同时具有多个身份。仅靠单一角色字段难以表达这些复杂权限关系。当前系统也没有统一后端权限拦截器或安全框架，若用户绕过前端页面直接调用接口，仍需要更多后端校验保证安全。后续改进方向是引入 RBAC 表结构，建立接口权限清单，并通过后端拦截器或 Spring Security 在服务端统一校验登录态和权限，从而减少对前端控制的依赖。

## 6.2.2 密码安全性仍可进一步增强

当前系统面向课程设计演示，密码处理以简单校验为主，数据库字段预留了 `VARCHAR(100)` 长度，但样例数据中的密码仍采用易读形式。这样的设计便于本地测试和演示账号登录，但不符合生产系统对密码安全的要求。真实系统中，密码不应以明文或可逆方式保存，也不应在日志、接口响应或页面中暴露。登录失败次数、账号锁定、密码强度、密码重置和会话失效策略也应纳入安全设计。

密码安全不足会带来较大风险。教务系统保存学生、教师、课程和成绩等敏感信息，一旦账号密码泄露，可能导致越权查看或篡改数据。后续改进应至少引入 BCrypt、PBKDF2 或 Argon2 等不可逆哈希算法，并为每个密码使用随机盐。登录接口应避免区分“账号不存在”和“密码错误”过于细致的提示，防止账号枚举；会话应设置合理过期时间；重要操作可要求二次确认或记录更详细审计日志。在课程设计报告中说明这一不足，体现了对演示系统和生产系统差异的认识。

同时，密码安全还应与用户管理流程配合，例如初始密码强制修改、密码复杂度校验、管理员重置密码留痕等。只有认证、存储和管理流程同时改进，账号安全性才会明显提升。

## 6.2.3 选课高并发控制尚未引入 Redis 或分布式锁

当前选课模块能够完成容量判断、重复选课判断、选课记录写入和已选人数更新，适合本地演示和低并发教学场景。然而，在真实高校选课高峰期，可能有大量学生同时选择同一门热门课程。如果多个请求同时读取到相同的剩余容量，再分别写入选课记录和更新已选人数，就可能出现超选、人数不一致或响应延迟等问题。当前系统尚未引入 Redis 原子计数、消息队列削峰、数据库条件更新、乐观锁、悲观锁或分布式锁等高并发控制方案。

这一不足并不影响课程设计基本功能，但限制了系统向真实生产环境扩展。后续改进可从三个层面进行：数据库层可采用条件更新或行级锁，保证只有在已选人数小于容量时才更新成功；缓存层可使用 Redis 维护课程剩余容量并进行原子扣减；架构层可通过消息队列异步处理选课请求，缓解瞬时压力。同时需要设计失败补偿和数据校准机制，确保缓存与数据库最终一致。报告中说明这一点，可以体现对选课业务复杂性的认识，即选课不只是插入一条记录，还涉及并发容量和一致性问题。

如果后续进行生产化改造，还应增加并发压测用例，模拟大量学生同时抢同一课程，从响应时间、失败率和最终人数三个角度验证控制方案是否可靠。

## 6.2.4 排课冲突检测算法仍较简化

系统任课任务中保存了学期、星期、开始节次、结束节次、周次和教室等信息，能够表达基本上课安排。当前项目已具备一定时间信息校验基础，但排课冲突检测仍较简化，尚未完整覆盖真实排课中的所有复杂情况。例如同一教师在同一学期、同一星期、相同节次不能承担多门课程；同一教室同一时间不能安排多个课程；学生选课时不同课程之间不能有节次冲突；周次表达如果包含单双周、连续周和不连续周，还需要更复杂的解析算法。

排课冲突检测的难点在于时间区间和周次集合的交叉判断。简单比较星期和节次只能覆盖一部分情况，若课程 A 在 1-8 周上课，课程 B 在 9-16 周上课，即使星期和节次相同也不冲突；若课程 A 在单周上课，课程 B 在双周上课，也不应被判定为冲突。后续可以把周次字符串解析为集合或位图，再结合星期、节次和教室进行交集判断。也可以把排课规则抽象为独立服务，使管理员创建任课任务和学生选课时复用同一冲突检测逻辑。当前系统的简化实现适合课程设计，但仍有明显算法扩展空间。

此外，真实排课还会受到教师偏好、教室容量、课程类型、实验室资源和行政班安排等因素影响，这些因素目前尚未进入系统规则模型。

## 6.2.5 GPA 规则尚未覆盖补考、重修等复杂教务场景

当前成绩模块以百分制成绩录入、成绩状态、平均分和已获学分统计为主。报告中设计的 GPA 折算可以基于百分制分段和学分加权实现，但当前系统尚未建立完整 GPA 规则表，也未覆盖补考、重修、缓考、免修、等级制课程、通过制课程、成绩作废和最高成绩计入等复杂教务场景。真实高校的绩点规则往往由学校制度决定，不同课程类型、不同考试性质和不同修读状态可能采用不同计算方式。

这一不足会影响成绩分析的精确性。若学生某门课程第一次不及格，重修后通过，系统需要决定是保留两次记录、取最高成绩、取最后成绩，还是同时影响绩点和学分；若课程为考查课或通过制课程，可能没有百分制成绩，但仍影响毕业要求；若学生补考通过，成绩可能按 60 分或实际补考分计入。当前系统的数据结构已经把成绩放在选课记录中，为扩展保留了基础，但仍需要增加考试类型、修读类型、成绩来源、是否计入 GPA 等字段或规则表。课程设计阶段可以先采用简化 GPA，但报告应明确其边界。

后续若要提高成绩模块的制度适配能力，应把绩点规则从固定算法中抽离出来，形成可配置规则，以便不同课程类型和不同年级方案使用不同计算口径。

## 6.2.6 系统尚未进行服务器部署和压力测试

当前系统主要面向本地开发和课程设计演示，前端、后端和数据库均在本地环境运行。虽然本地测试能够验证功能正确性，但还不能完全代表服务器部署后的稳定性。真实部署需要考虑前端构建产物托管、后端服务进程管理、数据库账号权限、跨域配置、反向代理、HTTPS、日志轮转、备份恢复和环境变量管理等问题。系统也尚未进行系统性压力测试，无法量化并发登录、并发选课、批量查询和报表统计在高负载下的响应时间。

缺少部署和压力测试意味着系统的性能边界尚不明确。报表聚合查询在数据量较小时运行良好，但当学生、课程和选课记录达到数十万级别时，索引设计、SQL 聚合和分页查询都可能成为瓶颈。前端页面在大量表格数据下也需要分页、虚拟滚动或筛选优化。后续可以将系统部署到服务器或容器环境，使用 JMeter、k6 或类似工具进行压力测试，记录吞吐量、响应时间、错误率和数据库负载。通过这些测试，才能从课程设计演示进一步走向可部署应用。

部署测试还应关注故障恢复能力，例如数据库备份恢复、日志轮转、配置隔离和服务重启后的会话处理。这些内容虽超出课程设计重点，但决定系统能否长期稳定运行。
