# 6.3 收获与体会

本节总结课程设计过程中的实践收获，重点围绕 Java Web 全栈流程、数据库设计、分层思想、权限异常接口联调和复杂业务闭环设计能力展开。

## 6.3.1 对 Java Web 全栈开发流程的理解

通过本次教务管理系统课程设计，可以较完整地理解 Java Web 全栈开发流程。项目不是只编写某一个接口或某一个页面，而是从需求分析开始，逐步完成数据库建模、后端分层、接口设计、前端页面、联调测试、异常处理和报告撰写。后端使用 Spring Boot 提供应用启动、依赖注入、Web 控制器和数据库访问能力；前端使用 Vue 3 构建页面和交互；数据库使用 MySQL 保存核心业务数据；接口通过 JSON 在前后端之间传递。各部分相互依赖，任何一处设计不清晰都会影响整体效果。

全栈开发的体会在于必须同时理解“页面需要什么”和“数据从哪里来”。例如学生选课页面需要展示课程名称、教师、学期、容量和已选状态，这些数据不是单表即可得到，而是需要任课任务、课程、教师和学生选课记录关联查询。教师成绩页面需要按教师身份过滤学生选课记录，学生成绩页面又需要从同一张选课成绩表中读取结果。通过这些实践，可以认识到 Web 系统开发不是前端和后端各自孤立完成，而是围绕业务流程持续对齐数据结构、接口字段和页面交互。该项目使我对 Java Web 应用从开发到运行的整体链路有了更具体的认识。

## 6.3.2 对数据库逻辑结构与物理结构设计的理解

本次课程设计加深了对数据库逻辑结构和物理结构的理解。逻辑结构关注实体和实体之间的关系，例如学院与专业、专业与班级、班级与学生、教研室与教师、教师与课程、学生与任课任务等。物理结构则需要把这些关系落实为表、字段、主键、外键、唯一约束、检查约束、状态字段和索引。仅仅列出表名并不能保证系统可用，关键在于每个字段是否有明确含义，每个关系是否有合理连接方式，每个约束是否能防止错误数据。

设计过程中最重要的体会是避免冗余和找准关系表。学生不直接保存学院和专业，而是通过班级逐级关联，可以减少信息不一致；教师与课程之间通过任课任务关联，可以表达不同学期和不同容量；学生与任课任务之间通过选课成绩表关联，可以同时保存选课状态和成绩。成绩不应放在学生表中，因为一个学生有多门课程；也不应放在课程表中，因为一门课程对应多个学生。通过这种分析，可以认识到数据库设计直接影响后续业务实现难度。好的表结构能够让业务流程自然展开，差的表结构会导致大量补救逻辑。

同时，物理字段类型也需要与业务含义匹配。例如学分适合使用定点小数，成绩需要允许空值以表示未录入，状态字段需要用明确取值表达正常、停用、已退课或已结课。

## 6.3.3 对 Controller / Service / Dao 分层思想的理解

本次项目实践使我更加理解 Controller、Service、Dao 分层思想。Controller 层面向外部请求，主要处理路径、参数和返回结果，不应直接写复杂 SQL；Service 层承载业务规则，例如字段必填、重复判断、容量判断、成绩范围、关联限制和事务边界；Dao 层负责和数据库交互，集中维护查询、插入、更新和统计逻辑。分层之后，程序结构更容易阅读，也更容易定位问题。接口失败时，可以先判断是请求参数问题、业务规则问题还是数据库访问问题。

分层思想还帮助测试更加清晰。Controller 测试关注接口能否被正确调用，Service 测试关注业务规则是否正确，Dao 测试关注数据库读写是否正确。基础模块采用一类一服务、一类一数据访问的方式，结构清楚；综合业务模块虽然集中在一个服务类和一个数据访问类中，但仍然遵循控制器不直接访问数据库的原则。通过本项目也可以体会到分层并不是机械增加类数量，而是为了降低耦合和提高可维护性。对于较复杂的选课、成绩和报表业务，后续可以继续按照业务域拆分服务，使结构更加清晰。

## 6.3.4 对权限控制、异常处理与接口联调的实践体会

权限控制、异常处理和接口联调是本次课程设计中较有实践价值的部分。权限控制要求系统理解不同角色的业务边界，不能让学生使用管理员功能，也不能让教师随意修改非本人课程成绩。异常处理要求系统面对错误输入时给出明确反馈，而不是简单失败或暴露底层错误。接口联调则要求前端、后端和数据库三方字段一致，例如前端提交的学生编号、任课任务编号和成绩字段必须与后端接收逻辑和数据库结构一致。

实践中可以发现，很多问题不是算法复杂，而是接口契约不一致造成的。例如前端字段名与后端预期不一致会导致参数为空，后端返回结构不统一会导致前端难以提示错误，数据库约束提示不友好会影响用户理解。因此统一返回结构、统一异常处理和清晰接口文档非常重要。权限和异常也不能只在页面上处理，后端仍需进行关键校验，以避免绕过前端造成数据风险。通过本项目，我认识到一个可用系统必须同时考虑正常路径和失败路径，接口联调的质量直接影响最终用户体验。

这也让我认识到调试时需要沿着“页面输入、网络请求、控制器参数、服务校验、数据库结果、前端提示”逐步排查，而不能只停留在页面现象本身。

## 6.3.5 对复杂业务闭环设计能力的提升

教务管理系统的难点不只是增删改查，而是多个模块之间形成业务闭环。管理员维护基础数据只是起点，任课任务把教师和课程连接起来，学生选课把学生和任课任务连接起来，教师成绩录入把教学结果写回选课记录，学生成绩查询和管理员报表又把这些结果展示出来。每个环节都依赖前一个环节的数据正确性，也会影响后一个环节的结果。通过实现和整理这个闭环，可以提升对复杂业务系统的整体设计能力。

闭环设计要求在一开始就考虑数据流向和约束。例如选课时必须考虑容量，因为容量影响学生是否能成功选择；退课时必须考虑已选人数回退，因为报表和可选课程页面依赖该数字；成绩录入必须考虑成绩范围和成绩状态，因为学生查询和成绩分布依赖这些字段；日志记录必须知道操作人和目标对象，才能进行追踪。通过这些实践，我认识到业务系统设计需要从整体流程出发，而不是孤立完成单个页面。该课程设计帮助我把数据库、后端、前端、测试和文档连接起来，对后续开发更复杂的信息管理系统具有直接借鉴意义。

这种闭环意识也会反过来影响编码习惯：每新增一个功能，都应同时思考它需要哪些数据、影响哪些统计、可能出现哪些异常，以及如何被测试验证。
