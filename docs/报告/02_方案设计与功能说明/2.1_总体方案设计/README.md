# 2.1 总体方案设计

本节说明系统的总体技术方案。系统以浏览器为用户入口，以 Spring Boot 后端提供 JSON 接口，以 MySQL 保存教务业务数据，以 JdbcTemplate 完成数据库访问。整体设计强调前后端职责分离、后端分层清晰、数据结构稳定、接口返回统一、异常响应可控，并通过角色字段和前端路由守卫形成基础权限边界。

## 2.1.1 B/S 前后端分离架构方案

本系统采用 B/S 架构，即用户通过浏览器访问前端页面，前端再通过 HTTP 请求访问后端接口。与传统将页面渲染和业务逻辑集中在服务器端的方式相比，前后端分离可以使页面交互、业务处理和数据持久化边界更加清晰。当前项目的前端工程位于 `frontend` 目录，使用 Vue 3、Element Plus、Vue Router、Axios 与 ECharts 构建页面、表单、表格、路由和图表；后端工程位于 `src/main/java` 目录，使用 Spring Boot 4.1.0、Java 17 与 Maven 组织控制器、服务层、数据访问层和通用异常处理。浏览器端不直接连接数据库，而是通过 Axios 调用 `/edu/api` 下的后端接口，后端完成权限判断、参数校验、业务处理和数据库访问后，再以统一 JSON 结果返回前端。

从职责划分看，前端负责用户界面、表单输入、角色菜单、页面跳转、全局消息提示和图表渲染；后端负责登录状态、业务规则、数据一致性、异常归一化和操作日志记录；数据库负责持久化保存学院、专业、班级、学生、教师、课程、任课任务、选课成绩和日志等核心数据。这样的结构使系统具有较好的可维护性：当页面布局需要调整时，通常只修改 Vue 页面和接口封装；当业务规则需要变化时，主要修改 Service 或 Dao 层；当数据结构变化时，则集中在 SQL 脚本、实体对象和数据访问逻辑中调整。课程设计场景下，该方案既能展示 Java Web 后端能力，也能体现现代 Web 前端交互能力，能够覆盖登录、基础资料维护、选课、成绩、统计报表等完整教务闭环。

## 2.1.2 MVC 与分层开发设计

系统后端按照 MVC 与分层开发思想组织。Controller 层面向 HTTP 请求，负责接收参数、调用业务服务并返回统一结果；Service 层负责教务业务规则，例如新增学生时的必填字段判断、选课时的容量判断、成绩录入时的分数范围校验、删除或停用时的数据状态控制；Dao 层负责通过 JdbcTemplate 执行数据库查询和更新，将 SQL 访问集中封装，避免控制器直接操作数据库。实体对象、DTO 和 VO 用于表达不同层之间的数据承载关系，通用返回对象、分页对象和异常类用于统一接口风格。当前项目中，学院、专业、教研室、班级、课程等基础模块已经形成 Controller、Service、Dao、Entity 的分层结构；用户、学生、教师、任课任务、选课成绩、报表等综合业务主要由 `AcademicBusinessService` 与 `AcademicBusinessDao` 承担业务与数据访问职责。

这种分层方式的核心价值在于降低耦合。若接口路径或页面调用方式发生变化，只需调整 Controller 或前端 API 封装，不应影响数据库访问细节；若数据库字段或查询方式发生调整，应优先在 Dao 层集中处理，不应把 SQL 分散在页面或控制器中；若业务规则发生变化，例如选课容量规则、成绩录入规则或停用状态规则，则应放在 Service 层集中维护。报告目录中提到的 `AuthService`、`GradeService`、`GradeDao` 等类可以理解为进一步细化后的理想分层职责，当前项目则采用相对集中的综合业务服务承载这些逻辑。该选择适合课程设计阶段快速完成闭环功能，但在后续扩展时，可以按认证、选课、成绩、报表等业务域继续拆分服务类，以提升大型项目中的模块独立性和测试便利性。

## 2.1.3 数据库访问方案（Spring JDBC / JdbcTemplate）

系统采用 Spring JDBC 与 JdbcTemplate 作为数据库访问方案。与直接使用原生 JDBC 相比，JdbcTemplate 封装了连接获取、语句执行、结果遍历和异常转换等重复性工作，能够减少样板代码，使开发者把主要精力集中在 SQL 语义和业务规则上。与重量级 ORM 框架相比，JdbcTemplate 更直观，SQL 可控性更强，适合课程设计中展示数据库表结构、关联查询、统计聚合和事务边界。当前项目的 `pom.xml` 引入了 Spring Boot JDBC 相关依赖，并通过 `application.yaml` 配置 MySQL 连接信息，数据库名称为 `edu_management_system`，主要建表脚本位于 `sql/02_create_tables.sql`。

数据访问层的设计以业务表为中心。基础信息表如 `college`、`major`、`department`、`class_info` 和 `course` 分别具有对应 Dao，用于完成列表查询、主键查询、新增、更新和状态停用。综合业务表如 `sys_user`、`student`、`teacher`、`teaching_task`、`student_course`、`teacher_student` 以及报表统计查询，由 `AcademicBusinessDao` 集中处理。日志表 `operation_log` 由 `OperationLogDao` 管理，支持记录操作信息和分页筛选查询。JdbcTemplate 返回的数据在部分综合接口中以键值映射形式承载，便于把跨表关联结果直接返回给前端表格；基础实体则以 Java 对象形式表达，便于明确字段含义。此方案在课程设计中具有较强的透明性：数据库表字段、外键关系、唯一约束、状态字段和统计查询都可以在报告中直接说明，同时避免在报告中展示具体 SQL 或源代码。

## 2.1.4 前后端接口通信与统一返回格式设计

前后端通信采用 HTTP 加 JSON 的方式。前端通过 Axios 封装各业务模块的 API，例如认证、学院、专业、教研室、班级、课程、学生、教师、任课任务、选课成绩、报表和操作日志等接口文件；后端控制器按照模块暴露 REST 风格路径，完成列表、详情、新增、修改、停用、选课、退课、成绩录入和报表查询等操作。系统后端设置了上下文路径 `/edu`，因此前端联调时需要以该上下文为基础访问接口。接口数据不依赖服务器端模板渲染，而是以结构化 JSON 在前后端之间传递，前端收到结果后再渲染为表格、对话框、标签、统计卡片或 ECharts 图表。

统一返回格式是接口通信稳定性的基础。当前项目使用 `Result` 承载响应状态，通常包含状态码、提示信息和业务数据。成功响应返回业务数据或空数据，失败响应返回明确错误信息，未登录、无权限、参数错误、未找到和服务器错误等情况通过不同状态语义区分。分页日志查询使用 `PageResult` 表示记录列表、总数、当前页和每页数量，方便前端分页组件消费。统一返回格式的优点在于前端无需针对每个接口单独猜测响应结构，可以在 Axios 响应拦截器或页面请求方法中统一判断状态并展示消息。对课程设计而言，这种设计也便于测试：接口测试只需要关注输入参数、响应状态、提示语和数据结构是否符合预期，而不必分析后端异常堆栈或数据库内部错误。

## 2.1.5 权限控制与异常处理总体方案

系统权限控制采用基础角色模型，登录用户表 `sys_user` 中保存 `ADMIN`、`TEACHER`、`STUDENT` 三类角色。管理员面向基础信息维护、课程与任课安排、统计报表和日志查询；教师面向本人任课课程、学生名单和成绩录入；学生面向个人课程选择、退课和成绩查询。当前项目后端通过登录接口建立会话状态，并在返回的登录用户信息中包含角色和用户标识；前端路由根据角色控制页面入口和菜单显示，避免普通学生直接进入管理员页面。目录中提到的 `AuthInterceptor` 与 `WebConfig` 可作为后续增强方案，用于在后端统一拦截未登录请求和角色越权请求；当前报告应如实说明，现阶段权限边界主要由认证接口、会话信息、前端路由守卫和业务接口参数共同支撑。

异常处理方面，系统设置了业务异常和全局异常处理机制。业务规则错误，例如用户名或密码错误、必填参数缺失、重复数据、课程容量已满、成绩不在合法范围、要操作的记录不存在等，应优先在 Service 层识别，并转换为业务异常或统一失败结果。数据库访问异常、参数解析异常和运行时异常则由全局异常处理类归一化为稳定的 `Result` 响应，避免把底层异常细节暴露给前端。前端收到失败响应后，通过 Element Plus 的消息提示向用户显示可理解的信息。该方案虽然尚未扩展为完整 RBAC 权限模型，但已经具备课程设计所需的基础安全边界和异常闭环：用户操作被角色约束，业务错误被明确提示，系统错误被统一兜底，日志模块可以记录关键操作以支持追踪。
