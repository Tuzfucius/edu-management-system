# 2.2 教务核心功能模块设计

本节围绕教务管理系统的核心业务闭环展开。系统以登录认证为入口，以基础信息管理为数据基础，以任课任务连接教师与课程，以学生选课连接学生与开课任务，以成绩录入和 GPA 折算形成教学评价，以报表统计和日志追踪形成管理反馈。

## 2.2.1 登录与角色权限控制模块

登录与角色权限控制模块是系统业务入口。用户通过登录页面输入账号和密码，前端调用认证接口，后端根据 `sys_user` 表校验账号是否存在、密码是否匹配、账号状态是否正常。校验通过后，后端返回用户编号、账号、角色等登录用户信息，并更新最后登录时间，同时可记录登录操作日志。前端保存登录态后，根据角色跳转到管理员、教师或学生对应页面。管理员主要进入基础信息、课程管理、报表统计和日志页面；教师进入任课课程、成绩录入和教师端报表页面；学生进入可选课程、我的成绩和个人看板页面。这样设计可以在用户进入系统的第一步就完成角色分流，减少无关页面暴露。

权限控制分为页面级和业务级两个层次。页面级控制由前端路由和布局菜单承担，根据登录用户角色决定哪些路由可进入、哪些导航项可显示。业务级控制则由后端服务逻辑承担，例如教师只能针对本人任课任务下的学生成绩进行录入，学生选课和退课必须基于本人的学生编号，管理员才能维护基础资料和查看全局日志。当前项目尚未把所有权限规则抽象为独立的后端拦截器或 RBAC 权限表，因此报告中应将其描述为“基础角色权限控制”，并说明后续可以通过 `AuthInterceptor`、`WebConfig`、角色权限表和接口权限清单进一步增强。该模块的核心目标不是追求复杂权限体系，而是保证三类用户在课程设计要求下具有清晰边界，能够支撑后续选课、成绩和报表功能安全运行。

## 2.2.2 基础信息管理模块（学院 / 专业 / 班级 / 教研室 / 学生 / 教师 / 课程 / 用户账号）

基础信息管理模块为整个教务系统提供基础主数据。学院、专业、班级构成学生所属组织链路，学院、教研室、教师构成教师所属组织链路，课程表保存课程编号、课程名称、学分、学时、课程类型和考核方式，用户账号表保存登录账号、密码、角色和状态。管理员通过基础信息页面维护这些数据，包括新增、修改、列表查询和停用。系统采用状态字段进行逻辑停用，而不是直接删除重要业务记录，原因在于学生、教师、课程一旦参与选课、成绩或任课任务，直接删除会破坏历史数据和统计结果。

该模块的设计重点在于数据一致性。专业必须隶属于已存在的学院，班级必须隶属于已存在的专业，教师必须隶属于已存在的教研室，学生必须隶属于已存在的班级并绑定用户账号，课程编号、学号、工号和登录账号应保持唯一。通过这种主从关系，可以避免学生表中同时保存专业和班级导致的矛盾，也可以避免教师信息脱离组织结构。前端页面通过 Element Plus 表格、表单和对话框呈现维护入口，后端通过对应 Controller、Service、Dao 处理请求，数据库通过唯一约束、外键约束和状态字段保证基础数据可靠。基础信息管理虽然业务流程不如选课复杂，但它是所有后续功能的前提，只有基础数据规范，任课任务、学生选课、成绩录入和统计报表才能形成可信结果。

## 2.2.3 任课与开课任务管理模块

任课与开课任务管理模块用于把课程、教师、学期、上课时间、教室和容量组合成具体的开课记录。单独的课程表只描述课程本身，例如课程编号、课程名称、学分和考核方式，并不说明某一学期由哪位教师授课。通过 `teaching_task` 表建立任课安排后，同一门课程可以在不同学期由不同教师开设，也可以由同一教师承担多门课程。该设计解决了教师与课程之间的多对多关系，并为学生选课、教师查看课程、成绩录入和教学工作量统计提供统一依据。

任课任务的关键字段包括课程、教师、学期、星期、节次、周次、教室、课程容量、已选人数和任务状态。管理员创建任务时需要选择有效课程和有效教师，并填写开课学期与容量。容量字段直接影响学生选课逻辑，已选人数字段用于快速判断剩余名额，任务状态用于区分可选、停用或已结课。当前项目对排课冲突的检测相对简化，主要保证数据结构能够表达时间地点信息；后续若扩展，可增加同一教师同一节次冲突、同一教室同一时间冲突和学生课表冲突检测。任课任务模块位于基础信息与业务流转之间，是连接“课程资源”和“教学执行”的桥梁。没有任课任务，学生不能直接选择抽象课程，教师也无法定位自己需要管理的具体班课。

## 2.2.4 学生自主选课与退课模块

学生自主选课与退课模块体现教务系统的核心交互。学生登录后，前端根据学生身份查询可选课程列表，可选课程来自任课任务及其关联课程、教师和容量信息。学生选择课程时，后端需要校验学生是否存在、任课任务是否可选、是否已经选过该任务、课程容量是否已经达到上限。校验通过后，在 `student_course` 表中创建选课记录，并同步增加任课任务的已选人数。若学生退课，则将选课记录状态置为退课，并减少对应任务的已选人数。由于 `student_course` 表对学生和任课任务建立唯一约束，系统可以从数据库层面防止同一学生重复选择同一开课任务。

该模块的关键在于容量判定和状态维护。容量判定不是简单看课程表，而是看具体任课任务的 `capacity` 和 `selected_count`。退课也不应直接删除记录，因为选课历史可能与日志、统计或成绩相关，采用状态字段可以保留历史痕迹。当前课程设计阶段的并发控制主要依赖事务顺序和数据库约束，尚未引入 Redis、分布式锁或数据库行级锁的复杂方案，因此报告中应客观说明其适用于单机课程设计和低并发演示场景。在测试时，需要覆盖正常选课、重复选课、容量已满、退课后再选、已结课任务不可选等场景。通过这些规则，学生端能够形成“查看可选课程、提交选课、查看已选课程、执行退课”的完整闭环。

## 2.2.5 成绩录入与学分绩点折算模块

成绩录入模块面向教师角色。教师登录后查看本人承担的任课任务，并进入相应课程的学生名单，对已选学生录入或修改成绩。成绩实际保存在 `student_course` 表中，因为成绩属于某个学生对某个任课任务的学习结果，而不属于单独的学生或单独的课程。系统需要校验成绩是否在 0 到 100 分范围内，成绩录入后将成绩状态标记为已录入。学生端查询成绩时，可同时展示课程名称、教师、学期、学分、分数、是否通过、已获学分和平均成绩等信息。当前项目没有独立的 `GradeController`、`GradeService`、`GradeDao` 文件，成绩业务主要由 `StudentCourseController`、`AcademicBusinessService` 和 `AcademicBusinessDao` 承担。

GPA 折算属于成绩模块的展示与统计扩展。课程设计可采用常见百分制到绩点的分段规则，例如 90 分及以上折算 4.0，80 到 89 分折算 3.0 到 3.9，70 到 79 分折算 2.0 到 2.9，60 到 69 分折算 1.0 到 1.9，低于 60 分不获得绩点和学分。综合 GPA 应按照课程学分加权，而不是简单平均所有成绩。报告中应说明 GPA 规则是课程设计简化模型，尚未覆盖补考、重修、缓考、免修、等级制课程等复杂教务场景。成绩模块的意义在于把选课结果转化为教学评价结果，使学生端能够查询个人学习成果，教师端能够完成教学管理，管理员和报表模块能够分析成绩分布。

## 2.2.6 数据可视化与统计报表模块

数据可视化与统计报表模块用于把数据库中的业务记录转化为管理视图。当前项目后端设置 `ReportController`，通过服务层和数据访问层提供概览统计、学院学生人数、成绩分布、教师任课负载等报表数据；前端使用 ECharts 将这些数据渲染为柱状图、饼图或折线类图形，并结合统计卡片展示学生总数、教师总数、课程总数、选课记录数等指标。报表数据不单独存储为统计表，而是通过数据库聚合查询实时生成，这样可以避免统计表与业务表之间出现同步延迟或数据不一致。

报表模块的设计重点是指标口径清晰。学院学生人数应基于学生、班级、专业和学院之间的关联关系统计，且通常只统计正常状态学生；成绩分布应基于已录入成绩的选课记录，不应把未录入成绩计入分数区间；教师任课负载应基于任课任务统计，能够反映不同教师承担的课程数量或学生规模。前端图表应只负责展示，不应在页面中重新定义复杂统计规则，以免前后端口径不一致。该模块的价值不仅在于美化页面，更在于形成管理反馈：管理员可以快速观察学院规模、课程受欢迎程度、教师教学负载和成绩分布情况；教师也可以通过报表了解所授课程的成绩集中区间和待录入数量。对于课程设计报告，应强调 ECharts 是数据桥接和可视化工具，核心仍然是后端统计接口与数据库关联模型。

## 2.2.7 系统日志与操作追踪模块

系统日志与操作追踪模块用于记录关键业务操作，提升系统可审计性和问题追溯能力。当前数据库中设置 `operation_log` 表，字段包括操作用户、模块名称、操作类型、目标表、目标记录、操作描述、IP 地址和创建时间。后端通过 `OperationLogService` 和 `OperationLogDao` 完成日志写入与分页查询，管理员可以在前端日志页面按照模块、操作类型、用户和时间范围筛选查看。日志记录适合覆盖登录、退出、新增、修改、停用、选课、退课、成绩录入等关键操作。

日志模块并不直接改变业务流程，但对系统可靠性具有重要意义。教务数据涉及学生、教师、课程和成绩等敏感信息，一旦出现误操作，需要知道是谁在什么时间对哪个模块进行了什么操作。日志记录还可以辅助测试和演示，例如在完成一次学生选课后，管理员能够看到对应操作痕迹；在教师录入成绩后，可以通过日志确认操作发生时间和目标对象。当前项目日志记录以应用层主动调用为主，并没有使用切面统一拦截所有接口，因此报告中应说明其属于课程设计阶段的轻量日志方案。后续若扩展，可以使用 AOP 对控制器操作统一记录，并对敏感字段做脱敏处理。该模块与异常处理、权限控制共同构成系统治理能力，使系统不只是能完成业务操作，也能对操作过程进行基本追踪。

## 2.2.8 系统异常处理模块

系统异常处理模块用于把业务错误、参数错误、数据库错误和运行时错误转换为前端可以理解的统一响应。教务管理系统中常见异常包括登录账号不存在、密码错误、账号停用、参数缺失、业务编号重复、选课容量已满、重复选课、成绩越界、记录不存在、级联删除冲突和越权访问等。如果每个控制器自行处理异常，响应格式和提示语容易不一致，前端也难以统一展示。因此系统通过业务异常类和全局异常处理类集中处理错误，并返回统一 `Result` 结构。

异常处理的基本原则是分层识别、统一返回、友好提示。Service 层最了解业务规则，应在发现业务不满足时主动抛出或返回业务异常，例如成绩小于 0 或大于 100 时提示成绩范围错误，课程已满时提示容量不足。Dao 层发生数据库约束异常时，不应把底层异常直接展示给用户，而应在全局异常处理中转换为重复数据、关联冲突或服务器错误等可理解信息。前端 Axios 响应拦截器和页面请求逻辑则负责把错误信息通过 Element Plus 消息提示展示出来。该模块与日志模块配合后，可以做到“用户看到简洁提示，开发者可在日志中追踪原因”。对于课程设计而言，异常处理模块能体现系统完整性，说明程序不仅关注正常路径，也考虑了边界值、错误输入和数据约束冲突。
