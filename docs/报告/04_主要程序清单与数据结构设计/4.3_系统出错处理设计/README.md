# 4.3 系统出错处理设计

本节说明系统可能出现的错误类型、后端异常响应机制、前端异常提示机制和典型异常流程。错误处理设计的目标是使用户得到清晰提示，使前端得到统一结构，使后端日志保留排查线索，同时避免底层异常细节直接暴露给用户。

## 4.3.1 出错信息分类（登录异常 / 权限异常 / 参数异常 / 重复数据 / 容量超限 / 成绩越界 / 级联删除冲突）

系统出错信息可以分为认证类、权限类、参数类、数据约束类、业务规则类和系统运行类。认证类异常包括账号不存在、密码错误、账号停用、会话过期等，通常发生在登录或恢复登录状态阶段。权限类异常包括未登录访问受保护页面、学生访问管理员功能、教师操作非本人课程成绩等。参数类异常包括请求字段缺失、编号为空、数值类型转换失败、日期格式不合法和分页参数越界。数据约束类异常包括登录账号重复、学号重复、工号重复、课程编号重复、组合唯一约束冲突以及外键引用失败。业务规则类异常包括课程容量已满、重复选课、不可选任务仍被选择、退课记录不存在、成绩小于 0 或大于 100。系统运行类异常包括数据库连接失败、未预期空值、服务运行异常等。

分类处理的意义在于不同错误应给出不同响应。登录异常应提示用户重新输入或联系管理员；权限异常应提示无权访问并阻止继续操作；参数异常应提示检查输入；重复数据应指出业务编号已存在；容量超限应提示该课程已满；成绩越界应明确合法范围；级联删除冲突应提示该记录已被业务使用，建议停用而不是删除。通过这样的分类，系统错误处理不再是简单的“操作失败”，而是与教务业务含义绑定。测试时也应按照这些分类准备数据，确保每一类错误都有可复现的输入和可验证的输出。

## 4.3.2 后端异常响应机制（BusinessException / @RestControllerAdvice / 统一 Result 返回）

后端异常响应机制由业务异常、全局异常处理和统一返回对象组成。业务异常用于表达可预期的业务错误，例如重复选课、容量已满、成绩越界、记录不存在等。这类错误不是程序崩溃，而是用户操作不满足业务规则，应返回明确提示。全局异常处理类使用控制器增强机制统一捕获异常，包括业务异常、请求体解析异常、参数绑定异常、数据访问异常、运行时异常和其他兜底异常。捕获后统一转换为 `Result` 结构，使前端无论遇到哪类错误，都能按相同响应格式处理。

统一响应机制应避免泄露内部细节。数据库唯一约束冲突不应把数据库约束名直接显示给用户，而应转化为“编号已存在”或“重复数据”；外键删除限制不应显示底层 SQL 错误，而应提示“该数据已被关联业务使用，不能直接删除”；运行时异常应返回服务器错误提示，同时在后端日志中记录请求标识和异常信息。当前项目的日志配置将日志写入 `logs/edu-management-system.log`，全局异常处理可以配合日志定位问题。该机制使控制器保持简洁，服务层只需关注业务规则，异常响应由统一位置处理。对于课程设计报告，应强调后端异常处理不仅提高用户体验，也使接口测试具有确定性，因为失败路径同样有规范结构。

## 4.3.3 前端异常提示机制（Axios 响应拦截 / Element Plus 全局消息提示）

前端异常提示机制建立在 Axios 请求封装和 Element Plus 消息组件之上。前端 API 文件按业务模块封装请求，使页面无需关心底层请求细节。请求返回后，前端可以根据统一 `Result` 中的状态码和提示信息判断是否成功；若失败，则通过全局消息或页面局部提示向用户说明原因。对于未登录或会话失效，前端应跳转登录页；对于无权限，前端应跳转无权限页面或阻止操作；对于参数、重复数据、容量、成绩等业务错误，应保留当前页面，让用户修改输入后重新提交。

前端提示需要做到明确但不过度暴露。用户执行新增学生时，如果学号重复，应提示学号已存在，而不是显示后端异常类名；学生选课容量已满时，应提示课程名或当前任务不可选，而不是只说请求失败；教师录入成绩越界时，应提示合法范围，帮助教师立即修正。页面层面还应在提交按钮、表单校验和列表刷新之间形成闭环：操作成功后刷新数据，操作失败后保留用户输入并显示错误原因。当前项目使用 Element Plus 表格、表单、对话框和消息提示，适合实现这种交互。前端异常机制与后端统一响应互相依赖，只有后端返回稳定结构，前端才能以较少逻辑覆盖多数错误场景。

## 4.3.4 典型异常处理流程（重复选课 / 容量已满 / 成绩越界 / 越权访问）

重复选课流程中，学生提交选课请求后，后端根据学生编号和任课任务编号查询是否已有有效选课记录。若存在，则不再新增记录，也不增加已选人数，而是返回重复选课提示。容量已满流程中，后端读取任课任务容量和已选人数，当已选人数不小于容量时，直接返回容量已满，阻止写入选课记录。成绩越界流程中，教师提交成绩后，服务层检查成绩是否在 0 到 100 范围内；若小于 0、大于 100 或格式不合法，则返回成绩范围错误，不更新成绩状态。越权访问流程中，未登录用户应被要求登录，角色不匹配用户应被禁止访问，教师操作非本人课程成绩时应返回无权限或记录无效。

这些典型流程共同体现“先校验、后写入、失败不改变数据”的原则。重复选课和容量已满保护选课人数一致性，成绩越界保护成绩统计和 GPA 计算正确性，越权访问保护教务数据边界。测试这些流程时，需要设计明确输入和预期输出。例如准备一名学生已经选择某任课任务，再次提交相同任务应失败且已选人数不变；准备容量等于已选人数的任务，提交选课应失败；提交 101 分或 -1 分应失败；使用学生角色访问管理员数据维护接口应被拒绝或页面不可进入。通过这些异常测试，可以证明系统不仅正常路径可运行，边界和错误路径也具有可控行为。
